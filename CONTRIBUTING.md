# 🎉 项目贡献指南（Conventional Commits + pnpm + Husky 流程）

### 为了保证代码质量、提交历史清晰以及团队协作一致性，请遵守本指南。

本项目强制启用以下工具：

- pnpm（包管理器）
- Prettier（代码格式化）
- ESLint（代码质量检查）
- Stylelint（如项目启用样式校验）
- lint-staged（只检查本次提交涉及的文件）
- Commitlint（提交信息规范）
- Husky（Git hooks）

所有提交行为必须符合本文档规范，否则提交会被自动拒绝。

### 请在提交代码时遵循以下格式：

```
type(scope): 描述
```

示例：

```
feat(auth): 新增登录接口
fix(api): 修复返回状态码错误
docs(readme): 更新使用说明
refactor(user): 优化用户模块结构
```

### 🔖 允许的 type

| type     | 说明                                 |
| -------- | ------------------------------------ |
| feat     | 新功能                               |
| fix      | 问题修复                             |
| docs     | 文档修改                             |
| style    | 不影响逻辑的格式调整（空格、缩进等） |
| refactor | 代码重构（不影响行为）               |
| test     | 测试代码变更                         |
| chore    | 构建流程、依赖更新、脚本变更         |

### 可选 scope，例如：

```
auth / user / api / config / ui / build ...
```

### 📌 编写规则

- type 必须小写
- : 后面必须有空格
- 描述必须简洁明了，不以句号结尾
- scope 可选，但推荐使用
- 描述可使用中文或英文，但要统一团队习惯

### ❗ 常见错误示例

- 增加登录功能 ❌ 缺少 type
- feat 新增登录功能 ❌ 格式错误（缺少冒号）
- feat:新增登录功能 ❌ 冒号后缺空格
- Fix(api): 修复问题 ❌ type 必须小写
- update code ❌ 描述不明确

### 💡 提交建议

- 每次提交只做一件事，避免“混合提交”
- 提交前确保代码已经通过本地格式化/检查（如 lint-staged）
- 提交信息的描述用动词开头，如“新增”、“修复”、“更新”
